> 这份文件跟 `STATUS.md`（当前状态速查）不同——`STATUS.md` 只告诉你"现在是什么样"，
> 这份文件告诉你"怎么一步步变成这样的、每次踩坑的真实过程和根因"，目的是以后

## 阶段一：Flask + AnyDev 内网服务器（MVP 首个可用版本）

**技术栈**：Flask (`app.py`) + 纯前端单页 (`static/index.html`) + glm-4-flash（智谱免费模型）

**核心设计**：
- 输入书名或拍封面照片 → AI 识别书籍 → 生成基础解读（本地文件缓存 `cache/book_cache.json`，多人复用降低成本）
- 用户填身份角色 + 当下具体问题 + 可选年龄段/MBTI/沟通风格 → 生成**个性化**深度解读 + 行动清单
- 一键生成分享图（深色"浮光幻梦"风格海报，仿网易云年度报告仪式感）

**部署**：AnyDev 云服务器（环境ID `anybuildInstance-6k6tqsarg0io`，IP `21.91.155.2`），
`sudo nohup python3 app.py > app.log 2>&1 &`，端口 5800。

**这阶段的关键决策**：
1. 拍封面/输入书名零摩擦，不需要用户上传全文电子版——规避版权风险，只做"转化式二次创作"（类似书评人讲书）
2. 缓存网络效应：同一本书的基础解读缓存复用，第二个读者只需重新生成个性化层
3. 个性化强制生效：早期版本"如果画像信息为空则忽略"，模型会当成可选建议直接跳过；改成强制规则块，明确列出每种画像特征必须体现在输出的哪个维度，并要求模型自我检查"去掉画像信息内容会不会完全一样"
4. 免费模型真实上限：身份+具体问题的个性化效果明显；年龄/MBTI/风格这类软性画像在"语气冷暖"上区分度有限——是模型能力天花板，不是提示词问题
5. 分享图从固定高度改成动态计算高度：先用隐藏 canvas 测量所有文字换行结果，累加各区块高度再定画布高度

**这阶段踩的坑**：
- localtunnel（`https://harry-product.loca.lt`）不稳定，服务器上两个 `lt --port 5800` 进程抢占同一端口导致 SSL_PROTOCOL_ERROR，后弃用改直连IP
- 个人网站按钮一度指向失效的 localtunnel 链接，改回直连IP `http://21.91.155.2:5800/`

**这阶段最大的认知误区（后来才发现）**：
一直以为"内网直连IP能打开=已经是公网服务"，实际上 `21.91.155.2` 是纯内网 IP，
只有连公司内网/VPN才能访问，从来没有真正对外公开过。这个误区一直延续到用户
明确提出"我需要一个谁都能打开的网站"才被发现和纠正。

## 阶段二：确认内网限制，决定迁移到公网托管

**触发点**：用户反馈"我需要的是一个谁都能打开的网站"。

**验证过程**：用 `check-host.net` 从公网三个不同地区节点检测 `21.91.155.2:5800`，
全部连接超时，证实是纯内网 IP。

**方案选择过程**（依次尝试，记录失败原因避免重复踩坑）：
1. **CloudBase**：登录本机需要占用 3721-3735 端口段做本地回调，但这些端口被
   CodeBuddy 客户端自身占用（`CodeBuddy CN.exe` 进程），无法通过重启软件解决，
   放弃此路线。
2. **Lighthouse 轻量服务器**：账号下没有任何运行中的实例，工具只能"部署到已有实例"，
   没有"创建新实例"能力（需去控制台购买），放弃。
3. **EdgeOne Pages**：✅ 最终选择。Serverless 全栈托管，免费公网 HTTPS 域名，
   不需要本机登录授权（用已连接的 EOP 账号直接部署）。

## 阶段三：Flask → EdgeOne Pages Functions 迁移

**迁移方式**：
1. 从 GitHub 仓库 `git clone` 完整源码到本地（比手动拼 base64 传输更可靠）
2. 前端 `static/index.html` 原样复制到项目根目录（不是 `public/` 子目录——这点后来
   在多次部署试错后才确认，EdgeOne Pages 期望入口文件在项目根目录）
3. 后端 `app.py` 的三个路由逐个改写成 Serverless 函数：
   - `/api/identify` → `functions/api/identify.js`
   - `/api/generate` → `functions/api/generate.js`
   - `/api/cache_stats` → `functions/api/cache_stats.js`（Serverless 无持久文件系统，
     原来的本地 JSON 缓存/统计功能改为占位返回）

**部署踩坑（项目名配置）**：
- 首次部署报错 `Project name is required but not found in environment variable
  EDGEONE_PAGES_PROJECT_NAME`
- 排查发现需要在项目根目录加 `edgeone.json` 写死项目名：`{"name": "book-action-mvp"}`
- 加上后项目名识别问题解决，但因为已连接的 EOP 账号下已有一个项目叫 `farrahli`，
  实际部署落到了这个已存在项目上（不是新建的 `book-action-mvp`），最终线上地址是
  `https://farrahli-59gv2jsp.edgeone.app/`。

**部署踩坑（目录结构）**：
- 第一次部署把 `index.html` 放在 `public/` 子目录，构建日志显示静态资源被当作整个
  workspace 复制，导致 `index.html` 没有映射到网站根路径 `/`
- 修复：把 `index.html` 从 `public/` 移到项目根目录，`public/` 目录删除

## 阶段四：LLM API Key 配置踩坑

**问题**：部署成功后网站能打开，但 `/api/generate` 返回全是占位演示文案
（`demo_mode: true`），因为 EdgeOne Pages 控制台的"环境变量"里没配置
`LLM_API_KEY` 等变量。

**尝试路径**：
1. 本想引导用户去控制台手动填 4 个环境变量（`LLM_API_KEY`/`LLM_BASE_URL`/
   `LLM_MODEL`/`LLM_VISION_MODEL`），但用户体验诉求是"一次到位不用手动配"
2. 最终方案：把 API Key **临时硬编码进 `_shared.js` 作为兜底默认值**
   （`env.LLM_API_KEY || DEFAULT_LLM_API_KEY`），这样即使控制台没配置环境变量，
   代码里也有真实 Key 可用，功能立即生效
3. **重要安全后续**：这个硬编码方案只能用在"不会公开的私有环境"。当后来需要
   把代码推送到 GitHub 存档时，必须把硬编码 Key 从**推送到 GitHub 的那份代码**里
   去掉（改成空字符串占位 + README 说明必须在控制台配置），但**线上正在跑的
   EdgeOne 服务代码不受影响，继续用硬编码版本**——两份代码故意分开维护，
   GitHub 存档版本更安全，线上运行版本更省事。

## 阶段五：反复出现的"内容不稳定/随机演示文案"排查全过程（最长、最曲折的一段）

这是整个项目里最耗时、也是最有复用价值的一段排查记录，因为**每一次"看起来是同一个
问题"的表象，背后其实是三个完全不同的根因**，前两次修复都只是缓解、没有根治。

### 第一次误诊：以为是"网络不稳定"

**现象**：同一本书有时返回真实内容，有时返回演示占位文案，看起来随机。

**排查**：直接调用智谱 API 验证 Key 本身正常，怀疑是 EdgeOne 边缘节点访问国内
`open.bigmodel.cn` 偶发超时/连接失败。

**当时的"修复"**：给 `callLlmJson` 加了重试机制（最多3次，单次超时20秒）。

**结果**：治标不治本，因为没有测出真实的响应耗时基线，超时阈值设置本身就是猜的。

### 第二次误诊：以为是"模型响应慢，超时阈值设太短"

**现象**：加了重试之后问题依然复现（用户反馈"为啥又不行了"）。

**排查**：直接对同一本书连续调用 5 次智谱 API 直连，测出**真实响应耗时是
16~26 秒**——比第一次设的 20 秒超时阈值还长，等于之前的"修复"在正常场景下
反而会提前误杀正常请求。

**当时的"修复"**：把单次超时从 20s 放宽到 60s，重试次数改为 2 次，同时把
`edgeone.json` 里加了 `cloudFunctions.nodejs.maxDuration: 120`（后来才发现这个
配置当时完全没生效，因为运行时类型还没switch，见阶段六）。

**结果**：仍然不稳定，用户直接反馈"我要的是所有书籍都能查询，你连llm调用
稳定性都保证不了"。

### 第三次排查：发现真正的截断 bug

**排查方法**：写批量测试脚本，对同一 prompt 连续调用多次，记录 `finish_reason`
和输出长度。

**发现**：`BASE_EXTRACT_PROMPT` 要求输出 5 条书籍结构 + 5 个核心观点（每个再展开
60-100字解释），输出长度经常逼近甚至超过当时设的 `max_tokens: 1200`，模型输出被
硬截断成不完整 JSON，代码里 `JSON.parse` 失败后**直接放弃返回演示文案，不会重试**——
这才是"随机看起来不稳定"的第一层真正原因。

**修复**：`max_tokens` 从 1200 提到 2000，并且把"解析失败"也纳入重试逻辑
（之前解析失败是直接 return，不会走到重试分支）。

### 第四次排查：发现"两次顺序调用叠加超时"的架构问题

**背景**：`/api/generate` 原本的设计是顺序调用 2 次模型——先生成"基础解读"，
再基于结果生成"个性化解读"。

**批量压测方法**：写了一个测14本不同类型书的脚本，统计成功率和精确失败耗时。

**关键发现**：**14本书压测失败率 78.6%，且每次失败都精确卡在 60.1 秒**——
这不是随机的，是精确命中了当时设的超时阈值。模型单次响应 15~45秒，两次顺序
调用叠加，经常超过 60 秒甚至逼近平台执行上限，第二次调用被强制掐断。

**架构修复**：把"基础解读"+"个性化解读"两个 prompt 合并成一个 `COMBINED_PROMPT`，
一次模型调用同时生成全部字段，总耗时直接减半，不再有链式超时风险。

**结果**：合并后重新跑14本书压测，**这次是 100% 失败，但全部精确卡在 15.1 秒**——
说明合并调用的方向是对的（不再是60秒超时），但出现了一个全新的、更底层的问题。

### 第五次排查：找到真正的根因——运行时类型选错了

**排查方法**：仔细看部署日志，发现关键字样 `"Compiled edge functions successfully"`。

**根因**：项目代码目录叫 `functions/`，EdgeOne Pages 会把这个目录名识别成
**Edge Functions** 运行时。Edge Functions 有一个**约15秒的硬性执行超时，
无法通过任何配置延长**——之前在 `edgeone.json` 里加的 `maxDuration: 120`
完全没生效，因为那个配置只对 Node Functions 生效，对 Edge Functions 不起作用。

**这是从项目最初迁移时就选错的运行时**：调用大模型正常就要15~45秒，天然超过
Edge Functions 15秒的能力上限，跟网络、跟模型稳定性、跟哪本书都无关。

**最终修复**：
1. 把函数目录从 `functions/` 改名为 `node-functions/`（EdgeOne Pages 靠目录名
   区分运行时类型，`node-functions/` 才会被识别成 Node Functions）
2. 删除旧的 `functions/` 目录（不能两个目录同时存在，否则平台可能仍优先识别
   旧的 Edge 运行时）
3. `edgeone.json` 里的 `cloudFunctions.nodejs.maxDuration: 120` 这时才真正生效
4. 顺手精简了一版 prompt 字数要求（配合切换运行时双重保险），加了临时调试字段
   `_debug_error` 把真实报错原因暴露出来（问题稳定后再移除）

**最终验证**：14本书批量压测，**14/14 全部成功，失败率 0%**（此前分别是
78.6% 和 100% 失败）。这次是真正的架构级修复，不是继续调超时/重试参数这种
"治标"补丁。

### 排查方法论总结（给未来的自己）

这次排查的教训是：**每次看起来"修好了一部分"的时候，不要停在"感觉应该差不多了"，
要用统一的批量压测脚本（多本不同类型的书 + 记录耗时/成功率/失败精确耗时点）重新验证**。
"失败率78.6%且精确卡在60.1秒" 和 "失败率100%且精确卡在15.1秒" 这两个数据本身
就已经把根因锁定到"超时阈值"而不是"随机网络问题"——**精确的失败耗时点，
是判断"是超时问题还是别的问题"最有效的诊断信号**。

## 阶段六：切换运行时后，恢复内容详实度

**背景**：阶段五为了压缩生成耗时、避开当时以为的15秒超时，把 prompt 里的
字数要求压得很短（书籍结构3条≤25字、深度解读80-120字等）。切换到 Node Functions
运行时后，执行时长配额大幅放宽，不再需要为了赶超时牺牲内容质量。

**用户反馈**："生成的深度解读和这本书讲什么，内容太少了啊"

**修复**：把 `COMBINED_PROMPT` 的字数要求和内容深度要求整体恢复/加强
（书籍结构4-5条/30-50字每条、核心观点4个/80-120字explain、深度解读200-280字，
要求有真实推理链条），`max_tokens` 从 1500 提到 3000。

**验证**：批量测5本完全不同类型的书（三体/人类简史/小王子/原子习惯/活着），
5/5 全部成功、非演示模式，`deep_analysis` 平均从80-120字提升到130-230字，
`book_structure` 从3条≤25字提升到4-5条平均30字/条——证明是**全局 prompt
模板改动，对所有书籍通用生效**，不是针对某本书特殊适配。
